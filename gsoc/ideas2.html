Idea 2: Co-authorship Social network App

Description: Biological networks can be visualized and analyzed using Cytoscape. Often researchers want to go beyond the network of proteins or genes and also look at the inter-connectedness between colleagues and institutions. Who tends to publish together? What institutions are most collaborative? Are there inter-disciplinary connections in my institution?


Goal: Create a visual summary of how individuals are connected. An existing App currently queries PubMed directly or takes in a file with a set of publications (from scopus or incites) and builds a co-publication network where nodes are authors and edges the number of publications two authors share. Different sources have different depths of data, for instance PubMed, the most widely accessible of the three main sources, has citation counts but they are limited to only those articles cited in pubmed central and its institution data is limited to the first author. Given that PubMed is the most accessible source we want to expand its functionality to include computed institution data as well as better citation count approximations, and H-index calculation for individuals or subsets of papers.

This functionality can be applied to several data types outside academia, whenever relationships between people is of interest. An example is professional social networks (e.g. linked-in), where individuals are connected by who they have worked with in the past or the twitter-verse where individuals are connected when they each reference the same (#) hash tag. This project will start with support for co-publication networks and extend to other networks like twitter and linked-in as time permits. Additionally, we want to expand our co-authorship cytoscape App to include generic file formats to easily create the above networks.

Idea 3: User Interaction on Biomedical Linked Data

During the GSoC 2012, we developed SemScape, a plugin that allows the use of Cytoscape as a browser for Semantic Web (Linked Data) knowledge bases. We are currently in the process of updating libs and packaging the plugin for the Cytoscape App Store.

The idea for this year’s project is to go one step further and to allow the use of Cytoscape to visually define queries in SPARQL. The basic idea is that from a graph, a sub-graph can be defined where a few nodes or edges can be marked as variables (and eventually filters can be associated to them): this is equivalent to a (simple) SPARQL query. The user can then define queries in a simple and interactive way, without knowing the underlying syntax, that can be stored and applied on different data sources.

The project can then be extended in three directions, depending on the interest of the student:

  • allowing users to push such queries to a repositories and hence share them (the current plugin support updates of queries from remote repositories)

  • Implementing more complex queries (unions, negation, …)

  • Implementing heuristics for query-selection: in the current plugin, queries can be performed on demand, but they are also associated to specific actions. For instance, right-selecting a node of type protein can present as an option to expend the graph with the results of a query which is can include the specific protein as a value. This is possible through a simple markup through which queries are annotated by the user. Such markup could be included in the visual query/construction process and/or extracted from simple query patterns.


Some of the ideas in this project have been already prototyped, so we have a good understanding of the difficulty and timing required.

This project has the potential to make Cytoscape an ideal tool to navigate and query Linked Data resources (both in biomedical and, more generally, in Open Data domains). It also opens interesting possibilities in analyzing graphs from graph knowledge bases that can be represented as RDF (e.g.: dbpedia, freebase,...).


Language and Skills: Java/Maven. Knowledge of OSGi would be helpful. Knowledge of Semantic Web technologies would be a plus.

Idea 4: CentiScaPe for multiple networks analysis and graphical output enhancement

Background: CentiScaPe is a Cytoscape plugin that allows to compute network centrality parameters. It is the most complete Cytoscape plugin for network centralities calculating Degree, Average Shortest Path, Eccentricity, Closeness, Betweenness, Centroid, Stress, Radiality, Eigenvector, Edge Betweenness, Bridging Centrality, for directed and undirected networks. CentiScaPe allows identifying network nodes that are relevant from both experimental and topological viewpoints. CentiScaPe also provides a Boolean logic- based tool that allows easy characterization of nodes whose topological relevance depends on more than one centrality. Finally, different graphic outputs and the included description of biological significance for each computed centrality facilitate the analysis by the end users not expert in graph theory, thus allowing easy node categorization and experimental prioritization.


Goal: The new beta version of CentiScaPe provides also the possibilities to analyze multiple networks and to integrate the analysis with experimental data set. This implies that a user can automatically start a computation of several networks and waits for the results. Anyway, the output of the analysis of a lot of networks (say 100) is useless if not supported by a smart and user friendly system to analyze the results. The Idea is to implements some techniques used in software analysis to extract information from the results of multiple computation in order to compare different networks and to extract topological based biomarkers:

  • The main framework for the results analysis is the Abstract Interpretation theory, particularly the interval analysis (and eventually others) [1].

  • Some statistical parameters, automatic analysis of the output, and new kinds of graphical outputs that can help to extract information from a multiple networks centrality analysis should be provided. The tools should for example be able to answer to questions as “How does the centroid value of the protein A chenge in the submitted 100 different networks?” or “How does the betweenness of protein B change if we consider 30 different phosporilation values for each proteins in the network N (this means we have 30 different experimental data set for the same network that must be included in the computation)?”.

  • The tools should also provide the implementation of the integration of experimental data-sets for the eigenvector, bridging centrality and edge betweenness (they are now available for all the other centralities).

Idea 5: Improving Cytoscape.js Based Viewer for SBGN Process Description Diagrams with Better Layout and Advanced Complexity Management Operations

Background: Cytoscape.js is a JavaScript graph library for analysis and visualization of relational information [1]. The Systems Biology Graphical Notation (SBGN) is a visual language developed by the community with the intention of enabling scientists to represent networks, including models of cellular processes, in a standard, unambiguous way [2]. SBGN comprises three complementary languages suited to represent different types of pathways. SBGN Process Descriptions is a language that permits the description of the processes (e.g. biochemical reactions) taking place in a biological system and makes heavy use of compound structures (e.g. compartments and molecular complexes). Often times, SBGN PD networks are too large to visualize and details become obscured. Strategies to conceal the "unwanted" details of a network, allowing one to temporarily remove certain nodes and edges and subsequently reintroduce the removed details. Complexity management techniques in graph visualization such as filtering, hiding, folding, and nesting play an important role in effective analysis of biological networks as much as any other domain. Last year, support for compound structures has been added to Cytoscape.js core and an SBGN viewer named SBGNViz has been built [3]. Some complexity management operations have also been designed and implemented within this tool to hide unwanted details on demand without violating SBGN PD network invariants (e.g. all substrates and products of a process must be shown at any point).

Goal: This year, we'd like to improve this tool in certain ways. First, it'd be great to improve the layout implementation that supports compound graphs. Secondly, existing complexity management operations could be extended to include folding/unfolding. Notice however that this is not a straightforward topology change. For instance collapsing a compound node into a single "meta" node requires calculation of any inter-graph edges going into the compound node and representing them with "meta" edges in the collapsed network. Similarly, expanding a collapsed node back requires undoing it. The situation becomes cumbersome since arbitrary level of nesting is allowed and folding can take place at an arbitrary level.

[1] http://cytoscape.github.io/cytoscape.js/

[2] http://www.sbgn.org/

[3] https://code.google.com/p/sbgnviz-js/

Idea 11: A SPARQL query plugin for Pathvisio

Background: The semantic web is becoming a valuable resource for Pathway knowledge. The challenge still remain to smoothly integrate this knowledge into biological frameworks. With their SPARQL endpoints Uniprot (http://beta.sparql.uniprot.org/) or the EBI rdf Platform (https://www.ebi.ac.uk/rdf/) , biological data is becoming available as semantic web data. To use this knowledge one need not only to learn SPARQL queries, but also be prepared to convert the semantic data in different formats that comes with the various biological platforms. Using semantic web data in Pathvisio, we currently need to compose the appropriate SPARQL queries, get the data and write various parser to change the data in to the different data models.


Goal: To make a Pathvisio plugin that allows a pathway curator to write SPARQL queries against a variety of semantic web based pathways resources and include the obtained triples into the Pathway under scrutiny. The plugin should also provide an interface to SPARQL results for other Pathvisio plugins. Having such SPARQL interface still requires a pathway editor to become knowledgeable in writing SPARQL queries. The burden of having to write various parsers for semantic web data however, is taken away. The plugin should:

  • Provide a SPARQL query editor, from where the user could compose queries, which are then subsequently launched to the respective SPARQL endpoints.

  • Provide an interface that wrap specific SPARQL queries for other Pathvisio plugins.

  • Link to a centralized repository (e.g. Github) where SPARQL queries can be stored for future reference and others to use.

  • Link to a centralized index of relevant SPARQL endpoints. This index might again be stored on platforms like Github.

  • Import the resulting triples, which can then be linked to recognizable pathway components, or as data elements to use in Pathway analysis.

  • Make a RDF representation of a Pathway opened in Pathvisio. These triples are to be used as template for SPARQL queries. The user should be able to click on a triple which is then added to the WHERE block in the SPARQL query.


Potential students: This idea is for a student who has solid background in SPARQL queries, or the Semantic web in general, and who is looking into opening up the Semantic Web to specific knowledge domains (I.e. Biology and/or Medicine).

Idea 26: Optimize Semi-supervised Learning Algorithm for Genomic Datasets and Inject Prior Knowledge from Biological Pathways

Background: A major problem in bioinformatics is that the number of features collected by modern high-throughput technologies severely overwhelms the number of samples for which such features are available. In a typical dataset, it is not uncommon to have tens of thousands of molecular measurements, but only a few tens or hundreds of samples. This severe undersampling creates a significant challenge for researchers who want to mine such datasets for biological or clinical insight. For example, when looking to predict sensitivity of cancer cells to various compounds, it is not uncommon to come up with a collection of molecular features that are highly predictive on a given dataset, only to have the accuracy fail to generalize to new, previously unseen samples.

A natural way to approach the issue is to bring in additional samples to the dataset, e.g., other cancer cells from the same tissue in the compound sensitivity example above. However, the phenotypic information or "labels" for such samples may not be readily available. Semi-supervised learning is a branch of machine learning that deals with constructing predictive models using both labeled and unlabeled data. It is particularly well-suited to biological applications, because while there is typically only a small handful of labeled samples, the unlabeled data is aplenty. Furthermore, we can expect some shared underlying biology between labeled and unlabeled data, due the fact that molecular measurements capture some underlying activity of various signaling and metabolic pathways in a cell. Semi-supervised learning can tap into the vast amount of unlabeled data to uncover such pathways and then utilize the labeled samples to correlate these pathways to phenotypic outcomes.

Goal: To implement a specific semi-supervised learning algorithm and to work with the mentor on modifying this algorithm to tailor to biological data. The algorithm was originally proposed by Ando & Zhang (2005) and operates by constructing multiple predictive tasks from the unlabeled data and learning across all those tasks simultaneously. The intuition behind the algorithm is that, if a molecular feature X is highly correlated with the phenotype F, then learning to predict X is just as valuable as learning to predict F directly. The advantage of predicting X is the fact that it's observed across all data (both labeled and unlabeled), while the measurements of F are, by definition, available for the labeled data only. Continuing with the example above, if a compound targets the epidermal growth factor receptor (EGFR) in a cell, then predicting whether EGFR is highly expressed is a task that should be highly related to directly predicting the sensitivity of the cell to that compound.

To simultaneously learn across multiple tasks, Ando & Zhang's algorithm makes an assumption that the tasks share some underlying lower-dimensional representation. This provides a perfect place to inject prior biological knowledge about feature-feature relationships. Specifically, a sub-goal of the project is to modify the lower-dimensional constraints in the algorithm to align with known information about biological pathways, as taken from web resources, such as WikiPathways, BIOCARTA, PathwayCommons, etc.

The basic implementation of the algorithm is already in place. It remains to optimize the implementation for handling full-scale genomic datasets (~20k features and thousands of samples) by moving parts of the code from R to C / C++, and to implement the pathway-based component discussed above.
